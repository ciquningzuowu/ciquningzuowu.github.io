工具

概述： 赋予智能体执行特殊操作的能力，具有连续调用、并行调用、重试和错误处理、跨工具调用功能。智能体遵循ReAct（推理+行动）模式，在简短的推理步骤和有针对性的工具调用之间交替进行，并将由此产生的观察结果输入到后续决策中。 1.静态工具
使用@tool装饰器创建函数，在创建代理时定义，且整个过程中保持不变。工具函数的编写需要在函数首行添加“”“ tool prompt ”“” 进行描述来让调用工具的模型能够确切理解该工具的用处。

2.动态工具

概述： 动态工具指在运行时调整可用的工具集，而不是预先定义所有工具。初始化代理时，注册过多的工具模会使模型过载增加错误，注册过少则会限制功能。可通过身份验证状态、用户权限、功能标志、对话阶段等调整可用工具集。

处理方法：
通过 @wrap_tool_call 创建中间件进行工具添加，可添加提示词，除方便开发人员识别外，可以被高级中间件技术管理选择调用（注册管理调用中间件的agent）

（1）筛选预注册工具

  • 完成特定对话里程碑后启用高级工具
  • 根据用户偏好、应用商店中的功能标志筛选工具
  • 运行时上下文中的用户权限筛选

（2）运行时工具注册
模型已部署时的添加方法，与上似乎区别不大（个人观点）

（3）初始全部注册，使用中间件在调用时筛选runtime中的tool_call需要的工具进行保留

3.工具错误处理

概述： 可通过 @wrap_tool_call 装饰器创建中间件，提示词设置为当出现工具运行错误时调用，返回错误信息给代理进行处理。

系统提示

概述： 提示词，指示智能体身份功能等信息，如：“You are a helpful assistant for prefessional. Be concise and accurate.Accurately respond to absolutely certain information.”

1.基础格式
可接受 str 和 SystemMessage 对象（后者可以更好地控制提示结构）。

2.动态系统提示
@dynamic_prompt 可创建根据模型请求生成系统提示的中间件。例如根据用户身份，可添加f"{base_prompt} Provide detailed technical responses."或f"{base_prompt} Explain concepts simply and avoid jargon."，使用return返回最终的提示词。

结构化输出

1.常规方法
使用 ToolStrategy 修饰BaseModel对象生成一个输出结构对象，传入模型初始化时的response_format参数。

2.提供者策略
ProviderStrategy 使用模型提供商的原生化结构化输出功能，该方法较为可靠，但仅适用于支持原生结构化输出的提供商。

记忆

概述： 代理会通过 message 自动维护对话历史记录，存储在message中的信息可以视为智能体的短期记忆，也可以通过配置进行自定义状态方案。

1.通过中间件自定义状态（首选）
（1）定义继承AgentState类的对象，添加各种属性。（例如: user_preferences: dict,custom_context: any）
（2）使用创建继承AgentMiddleware对象的类，在其中创建AgentState实例为属性，包含其他属性，可重写before_model等方法，最后将其添加到模型初始化时的Middleware注册表。

2.更改state_schema参数
初始化代理时将设定好的AgentState对象作为state_schema参数的值。

流媒体

概述： 当代理程序执行多个步骤时需要一定时间，微课显示中间的进度，我们可以试试回传信息。

中间件

用法：

  • 在调用模型之前处理状态（例如，消息修剪、上下文注入）
  • 修改或验证模型的响应（例如，防护措施、内容过滤）
  • 使用自定义逻辑处理工具执行错误
  • 基于状态或上下文实现动态模型选择
  • 添加自定义日志记录、监控或分析功能

二、model（搞半天原来还要自己拼）

概述： 语言学习模型（LLM） 除文本生成外，还具有工具调用、结构化输出、多模态、推理。本部分为开发调试时所用技术。

基础用法

1.创建代理时动态指定模型
2.直接在代理循环之外调用模型执行任务

初始化模型

最简单的方法：init_chat_model("model_name")

参数

  • model：模型标识，不只是让你看，需要模糊匹配模型的名字以供识别对接什么模型
  • api_key: 你自己注册的端口，消耗你自己花钱买的token，一点也不重要，注册了先发给我（doge）
  • temperature: 模型输出的随机性，数值越高，响应的创造性越高（同时出错率也更高）
  • max_tokens: 限制模型响应时最大输出token量，防止token浪费
  • timeout: 等待响应时间限制
  • max_retries: 出现网络问题等时的最大重复尝试次数

调用

使用invoke传入dict对象（包含human_message等信息），传入时可以包含config设置本次调用的临时配置。

流式

调用.stream()会返回一个迭代器，他会在生成过程中实时输出数据块，可通过循环实时处理每个数据块。（这里的迭代器更接近async操作，不断重复进行获取直至响应输出结束）。

将获取的所有数据块按时间顺序组合即为本次对话得到的完整响应。

批

概述: 将一系列独立的模型请求批量处理，由于可以并行处理，可以显著提高性能并降低成本。调用方式与invoke、stream相同。最终同时返回多份响应，可用for进行遍历。 （可以使用 batch_as_completed 实现每个输入生成后立刻返回）

工具调用

概述： 要使你定义的工具可供模型使用，必须使用绑定方法 bind_tools 将它们绑定。该方法返回的是新的model对象，而不是向源对象添加工具，原因为langchain主要通过链式组合（例如在中间件中会临时添加工具和更改模型等）和配置叠加，如果直接修改源对象，可能导致链条断掉或混乱，且该方法只是复制了配置字典和工具的应用，创建新模型的代价并不大。

结构化输出

概述： 声明并编写response结构，使用model.with_structed_output (response_class,//add " method="json_schema", "when use JSON Schema ),该方法 返回的是新的model对象。

支持的模型：（可存在嵌套结构）

  • Pydantic中的 BaseModel模型类 ，功能丰富，包含字段验证、描述和嵌套结构。
  • typing_extensions中的 TypeDict类 ，相较Pydantic更简单，适合不需要运行时的验证。
  • JSON Schema 可以实现最大程度的控制和互操作性，不过存在编写相较前两者较麻烦的缺点

高级主题（感觉这里的翻译并不准确，原文为 Advanced topics ）

概述： LangChain的 .profile方法 支持定义一个功能和特性字典（可限定例如最大输入、是否允许图像数据、是否可调用工具等属性）

多模态

概述： 模型处理并返回非文本数据（images，audio，studio）

推理

概述： 许多模型能够进行多步骤推理得出结论。若底层模型支持，可以将这个推理过程展现出来，以便更好地理解模型是如何得出最终答案的。（.stream() 方法）

token监测

概述： 可注册上下文管理器 UsageMetadataCallbackHandler 对模型使用的token进行监测。

三、message

概述： 模型的基本上下文单元，代表模型的输入输出，鞋带与LLM交互时的获取对话状态所需的内容和元数据。

message是包含以下内容的对象：

  • 角色——system，user
  • 内容——消息的实际内容，text，image，studio等
  • 元数据——响应信息、消息ID、令牌使用状况等

基本用法

  1. 创建消息对象并在调用时传给模型。
  1. 字典格式传入

消息类型

  • 系统消息——告诉模型如何运行，并为交互提供上下文
  • 人类消息——代表用户输入以及与模型的交互
  • AI消息——模型生成的响应，包括文本内容、工具调用和元数据
  • 工具消息——表示工具调用的输出

  1. 系统消息（ SystemMessage ）

    初始指令，用于启动模型的行为，可以用于设定基调、定义模型的角色并建立相应准则。

  2. 人类信息（ HumanMessage）

    代表用户输入与交互。可以包含text、image、studio、file等多模态内容。包括常规文本并可包含消息元数据（例如用户名字id等）。

  3. 人工智能信息（ AIMessage）

    模型调用的输出，可包含多模态数据、工具调用和特定于提供商的元数据。提供商对消息类型的权重不同，这意味着又是手动创建一个新的AIMessage对象并插入消息历史记录中会很有帮助。

  4. 工具调用

    模型调用工具时，调用信息会包含在AIMessage中（调用响应体的 .tool_call属性 获取）。其他类似推理和引用的结构化数据也可以出现。除此之外，AIMessage中也可以保存令牌计数和其他使用元数据。（调用响应体的 .usage_metadata属性 获取）

    对于支持工具调用的模型，AIMessage可以包含工具信息。工具消息用于将单次工具执行的结果传递回模型。（个人不是太明白这种方式的意义）

消息内容

概述： 消息具有content弱类型属性，支持字符串和无类型对象列表（例如字典）。这使得Langchain能直接支持提供者原生结构，例如多模态内容。

标准内容块: LangChain为消息提供了一种跨提供商通用的标准表示形式，消息对象实现了一个content_block属性，该属性会将 content 特性解析为标准的、类型安全的表示形式。

四、short memory

概述: 短期记忆可以让应用程序记住单个线程或对话的先前交互，最常见的形式是对话历史。model通过message接受上下文信息，包括系统信息和用户信息。由于上下文窗口有限，许多应用可以通过一些技术来益处或者以往过时的信息。

基础使用

创建代理时指定checkpointer

生产环境

checkpointer为开发阶段使用的对象，只能在程序运行期间存储短期记忆，生产环境中使用PostgoresSaver，该对象可将上下文记忆存储进postgresql数据库.

自定义代理内存

概述： 默认情况下，代理使用密匙AgentState管理短期记忆，特别是对话历史记录message。

消息清理

  1. 修剪消息

要在代理中清理消息历史记录，需使用@before_model中间件装饰器，在调用代理前修剪消息。以下为保留最开始时的原始信息与最新的三条信息。

  1. 删除消息 使用RemoveMessage并指定要删除信息的id，或者要删除内容的指令，如REMOVE_ALL_MESSAGES为全部删除。

  2. 消息摘要 概述: 直接修剪或删除消息容易导致有效信息损失，因此需要采用更加复杂的方法。创建代理时向middleware参数中添加内置函数SummarizationMiddleware，可自动汇总消息历史记录。

访问内存

  1. 工具 （1）使用工具访问短期记忆： 工具函数的参数部分添加runtime:ToolRuntime。可在函数体中访问runtime的属性以获取例如config，state等的信息。 （2）从工具写入短期记忆： 可通过从工具返回状态更新的方式实现。
  1. 提示词 在中间件中访问短期记忆，需要在@`before_mode 阶段根据历史记录或自定义状态字段创建动态提示后返回新的系统提示词。

五、stream

概述： LangChain的流式系统允许将代理运行的实时反馈呈现给应用程序。具体有以下功能：

  1. Sream agent progress
  2. Stream LLM tokens
  3. Stream thinking/reasoning tokens
  4. **Stream custom updates
  5. Stream multiple models: 可混合选择updates（返回state的更新）、messages（LLM token+元数据，模型的输出内容）、custom（任意用户数据）

流式进程

使用stream 或 astream方法并选择stream_mode="updates(or other)",这将在代理执行每个步骤后发送一个事件。例：

  • LLM节点：带有工具调用请求的AIMessage
  • 工具节点：带有执行结果的ToolMessage
  • LLM节点：最终响应结果

LLM token

使用stream_mode="messages"后可实时传输LLM生成的token。可访问可以查看流式工具调用和最终响应。

Custom updates

要在运行时更改stream模式，可以使用get_stream_writer,当stream模式为custom时，writer会立刻将内容返回而不需要等待整个响应流程结束，这可以用来返回类似"程序正在思考中"等过渡信息，且这个操作是进步进行的，不会显著影响主进程的运行。

Stream multiple modes

上述三种模式可混合使用。

六、structured output

概述： 允许自定义JSON对象、Pydantic models对象或直接用数据类对象作为格式输出限制。使用方式为在create_agent的response_format参数注册，且需注意的是，响应对象格式声明可以使用Union，即：

1.原生策略

概述： 部分模型厂商支持通过API使用原生输出结构，一般来说，厂商提供的原生结构可使用的最可靠的方法。

使用：定义ProvideStrategy类，属性有下：

  1. schema：支持Pydantic models（返回pydantic实例）、Dataclasses（返回带类型注解的python类）、TyprdDict（返回字典）、Json Schema（返回字典）
  2. strict：是否启用严格的模式遵循，需注意仅部分厂商支持此功能

2.工具策略

概述：对于不支持原生结构输出的模型，可以使用工具调用实现相同作用。这支持左右的模型使用。

使用：定义ToolStrategy类,属性有下：

  1. schema：与原生策略大致相同，支持Pydantic models（返回pydantic实例）、Dataclasses（返回带类型注解的python类）、TyprdDict（返回字典）、Json Schema（返回字典），但额外可以使用Union types，模型会自动给予上下文选择最合适的结构。
  2. tool_message_content：调用时会返回的工具消息的定义，如果不提供，默认返回结构化形影数据。
  3. handle_errors：结构化输出验证失败的错误处理策略，默认为True
    • True：默认错误模板捕获所有错误
    • str：自定义消息捕获所有错误
    • type[exceotion]：仅使用默认消息捕获此异常类型
    • tuple[type[Exception]]：仅捕获这些类型的异常，并使用默认消息
    • Callable[[Exception],str]：返回错误消息的自定义函数
    • False：不重试，让异常传播
  4. Custom tool message content：支持自定义工具消息内容，当调用工具结构化输出时会出现在会话历史中。
  5. **handleing errors： Error Handing：可设置当发生结构输出应用错误时的重试机制。
    • Multiple structured outputs error：多重结构化输出错误，当模型错误的调用多个结构化输出工具时，代理会提供错误反馈。
    • Schema validation error：架构验证错误，当结构输出不匹配期望架构时，智能体会提供特定的错误反馈。自定义handle_errprs属性，

handle_errors参数可设置处理方式:

  • 一段让智能体识别怎么处理的字符串。
  • 仅处理一个或多个特定异常情况，仅在发生指定异常时会重试，其他所有情况下都会引发异常。
  • 自定义错误处理函数

七、middleware

概述：中间件提供了一种能更彻底的公职智能体内发行什么的方式，对一下场景较为有效：

  • 通过记录、分析和调试追溯agent行为
  • 改变提示词、工具选择以及输出格式
  • 添加重试、回退以及终止逻辑
  • 应用速率限制、防护措施和个人身份信息检测

1.代理循环

概述：最简单的模型运行流程包括调用模型，让模型选择要执行的工具，然后在不再需要调用任何工具时结束。Middleware运行在智能体调用前、模型调用前、模型调用后、智能体调用后四个阶段，以及工具运行时前后的整个阶段。

2.预构建中间键

概述：Langchain和Deep Agents（深度代理）提供了预创建的中间件以供日常情况使用，每个中间件都已准备投入生产并且为你的特定需要做贡献。

3.自定义中间件

Hook（拦截点）

概述：通过在代理执行流程的特定点运行拦截点（hook） 来构建自定义中间件。一下为两种运行agent时插入拦截点的方式：

（1）Node-style hooks： 在执行节点使用，用于日志记录、验证和更新 相关hook：

  • before_agent - Before agent starts (once per invocation)
  • before_model - Before each model call
  • after_model - After each model response
  • after_agent - After agent completes (once per invocation)

（2）Wrap-style hooks 拦截执行过程，并在调用处理程序时进行控制。可用于重试、缓存和转换 相关hook：

  • wrap_model_call 每次模型调用前后
  • wrap_tool_call 每次工具调用前后

两种方式均可更新代理状态，机制也有所不同

  • Node风格返回一个字典，使用图的reducer应用于代理状态。
  • 包装式拦截点对于模型调用返回ExtendedModelResponse和一个向模型响应中注入的命令。对于工具调用直接返回一个命令。 （3）包含多个中间件的组合： 当多个中间件返回ExtendedModelResponse,它们的命令组成如下：
  • 命令是通过reducer执行：每个命令都会触发一个单独的状态更新，对于message而言，它们是累加的
  • 对于非reducer状态字段，冲突时，外层优先
  • 可重试：如果外部中间件实现了可能导致多次调用handler()的逻辑，则丢弃先前调用的命令

创建中间件

（1）使用装饰器封装函数，可快速创建单个hook的中间件，下为可供选择的装饰器：

Node-style：

  • @before_agent- 在代理启动前运行（每次调用运行一次）
  • @before_model- 在每次模型调用之前运行
  • @after_model- 在每次模型响应后运行
  • @after_agent- 在代理程序完成后运行（每次调用一次） Wrap-style：
  • @wrap_model_call- 使用自定义逻辑包装每个模型调用
  • @wrap_tool_call- 使用自定义逻辑包装每个工具调用

Convenience：

  • @dynamic_prompt- 生成动态系统提示

（2）基于class的中间件，更适用于多个hook或配置的复杂中间件。可用于定义实现同步或异步实现的hook或在单个中间件组件中组合多个hook

自定义状态模式

中间件可以使用自定义属性扩展代理的状态，这使得中间件能够：

  • 跟踪执行过程中的状态：在生命周期中保持计数、标记和其他值的正常运行
  • 在中间件之间共享数据：在不同hook中间件之间传递数据
  • 应用横切关注点：不修改agent核心逻辑的情况下添加例如速率限制、情况追踪、用户上下文或审查日志等功能
  • 做出条件决策：利用累计状态确定继续执行、跳转不同节点还是动态地修改行为

执行命令

使用多个中间件时，其运行如下：

Before hooks run in order:

  1. middleware1.before_agent()
  2. middleware2.before_agent()
  3. middleware3.before_agent()

Agent loop starts

  1. middleware1.before_model()
  2. middleware2.before_model()
  3. middleware3.before_model()

Wrap hooks nest like function calls:

  1. middleware1.wrap_model_call() → middleware2.wrap_model_call() → middleware3.wrap_model_call() → model

After hooks run in reverse order:

  1. middleware3.after_model()
  2. middleware2.after_model()
  3. middleware1.after_model()

Agent loop ends

  1. middleware3.after_agent()
  2. middleware2.after_agent()
  3. middleware1.after_agent()

Agent jumps

为了提前结束中间件的运行，可以通过@hook_config添加一个伴随着jump_to的字典，可选jump标签有下：

  • 'end'：跳转到运行结束后（或第一个after_agenthook后）
  • 'tools'：跳转工具节点
  • 'model'：跳转model节点（或第一个before_modelhook之后）

中间件规范

  1. 保持中间件的专注，每个中间件都做好一件事
  2. 优雅地处理错误，不要让中间件错误导致代理崩溃
  3. 使用合适的hook类型：
  • 用于顺序逻辑（日志记录、验证）的节点式
  • 控制流（重试、回退、缓存）的包装式
  1. 清楚的记录所有自定义状态
  2. 在集成之前，对中间件进行独立的单元测试
  3. 考虑执行顺序，将关键的中间件放在列表的最前面
  4. 尽可能使用内置中间件

中间件的预定义参数

  • RunInput / Messages:
    • Definition: The list of chat messages (Human, AI, System) being sent to the agent.
    • Action: You can modify, add, or remove messages here. For example, injecting a system prompt dynamically.
  • ToolCalls:
    • Definition: Requests made by the LLM to execute a specific tool (function).
    • Action: You can validate arguments, mock a response, or block a tool call if it looks dangerous.
  • Context (or Runtime):
    • Definition: A dictionary-like object storing state information throughout the agent's execution.
    • Action: Middleware can read/write data here to share information between different steps of the agent's workflow without passing it explicitly in the chat.

十一、context engineering（上下文工程）

概述：构建智能体最难的地方在于确保其可靠性，开发环境有效而生产环境失效常常发生。上下文工程是指以正确的格式提供正确的信息和工具，以便语言学习模型能够完成任务。

代理推导失败的两个主要原因：

  • 底层LLM功能不足
  • 正确的上下文没有传递给LLM 往往导致问题的原因是第二个。

Agent微调点：

1.可控制的context：

  • Model Context：模型调用中包含哪些内容（指令、消息历史记录、工具、响应格式）
  • Tool Context：哪些工具可以访问和生成（state、store、io of runtime context）
  • Life-cycle Content：模型调用和工具调用之间会发生什么（摘要、防护措施、日志记录）

2.数据来源

  • runtime：用户信息、身份权限、数据库连接等
  • state：当前信息、已上传文件、身份验证状态
  • store：用户偏好、提取的记忆、历史数据

3.模型上下文

System prompt：决定LLM的行为和功能 Message：发送给LLm的提示信息，管理message至关重要 Tool：使模型能与数据库、API和外部系统进行交互，工具的定义与选择方式非常重要。定义工具是每个工具都需要清晰的名称、描述、参数名称和参数描述。过多的工具会使得模型上下文过载，过少则会限制功能，通过动态工具选择，调整可用工具集 Model：不同模型之间各有能力优势、成本和适用范围 output struct

十二、模型上下文协议（MCP）

概述：一种开放协议，规范了应用程序如何向语言学习模型（LLM）提供工具和上下文。LangChain可以使用langchain-mcp-adapters库使用MCP服务器上定义的工具。

访问多个mcp服务器：使用库中的MultiServerMCPClient对象创建客户端对象，在其中注入mcp服务器信息，使用await 进行工具获取，将生成的实例在creat_agent函数中注入。

自定义server

使用fastmcp库可进行自定义mcp server，mcp支持不同的客户端-服务器通信传输机制

1.http

langchain-mcp-adapters底层使用了官方的MCP SDK，允许通过实现httpx.Auth接口提供自定义身份验证机制

2.stdio

client以子进程方式启动服务器，并通过标准输入/输出进行通信，适合本地工具和简单配置。

3.stateful sessions

MultiServerMCPClient默认是无状态的，每次工具调用都会创建一个新的MCP会话，并在执行工具后清理。若需要控制MCP会话的生命周期，可以使用client.session()创建持久的ClientSession

核心功能：

tool：MCP服务器可公开可执行函数，供LLM调用执行操作。LangChain将MCP工具转换为LangChain工具，可以直接在代理和工作流中使用

返回：

  • 加载工具：使用client.get_tools()从mcp服务器检索工具并将其传递给代理
  • 返回结构化内容：当MCP工具返回Content，适配器会在MCPToolArtifact中清洗处理它并作为工具产物返回它。可以使用ToolMessage对象的artifact字段获取此数据
  • 多模态：MCP工具可以在响应中返回多模态内容，当MCP返回内容包含多种类型时，适配器会将其转换为LangChain的标准内容块content_blocks（dict），可以通过块的type键的值判断是何种类的数据。
  • 资源：MCP服务器具有资源功能，可以返回数据，使用client.get_resources("server")可以从MCP服务器中获取资源。另一方式为使用load_mcp_resources并注入一个session对象直接执行 。
  • 提示词：MCP服务器的提示词功能可以提供可重复利用的提示词模板，LangChain会将MCP提示词转换到message中。使用client.get_prompt()或load_mcp_prompt()可获取提示词数据。

高级功能

1.工具拦截器：

概述：MCP服务器作为独立进程运行，无法访问例如上下文、代理状态等运行信息，但可以在MCP工具执行期间通过拦截器访问运行上下文弥补这一点。

拦截器还提供了类似中间件的功能（MCPToolCallRequest），例如控制工具调用，包括修改请求、实现重试、动态添加表头，或者完全终止执行。通过MCPToolCallRequest的runtime字段，可以获取上下文内容。

状态更新和命令： 概述：拦截器可以返回Command对象来更新代理状态或控制图的执行流程，实现切换代理、提前结束执行(goto="__end__")等功能。

定制拦截器： 概述：拦截器是封装工具执行的异步函数，可以实现请求/响应修改、重试逻辑以及其他横切关注点，运行顺序遵循洋葱结构，列表中第一个拦截器位于最外层

基本方式： 拦截器接受一个请求和一个处理程序，可以在调用处理程序之前修改请求，在调用处理程序之后修改响应或跳过

修改请求 使用request.override()创建一个修改后的新请求(不会修改原始请求)。可以使用get_token_for_tool获取请求的上下文。

组合拦截器 多个拦截器按照"洋葱状"顺序组合，注入MultiSeverMCPClient,按照注入顺序构成组合拦截器

日志监控

概述：可监控长时间运行工具执行的进度更新信息

获取进程信息： 创建进程监控函数，注入CallbackContext对象(可通过该对象的server_name和tool_name属性获取正在运行的服务器名称和正在执行的工具名称)，然后将函数注入在MulyiServerMCPClient的callbacks方法的on_progress参数

记录日志： 创建日志记录函数，注入LoggingMessageNotificationParams对象，然后将函数注入 MulyiServerMCPClient的callbacks方法的on_logging_progress参数

主动与用户进行结构化交互

在mcp服务器工具中可以使用ctx.elicit从用户获取额外信息，该函数效果为暂停进程并向用户发送一个明确的数据请求，这可以用来获取初始传入信息中缺失的信息，例如注册函数只收到了名字没收到年龄，可发送一次对年龄数据的获取请求到客户端

可以注册回调函数处理ElicitRequestParams（elicit函数发送的请求对象）,通过返回 ElicitResult对象处理ElicitRequestParams请求,将函数注入MulyiServerMCPClient的callbacks方法的on_elicitation参数

  3. description：自定义描述

响应中断

概述：智能体进程会在任务结束或者引发中断时停止，可以在interrupt_on中设置中断条件，当工具调用满足条件时便会触发中断。在version="v2"的情况下，会发生GraphOutput并且一个包含了待审查操作的interrupts对象会被输出。可将该对象提交给审核人员，然后根据审查结果恢复执行。

流式人机交互(LangGraph内容)

概述：通过使用stream()可以在agent运行以及处理终端时获取实时的最新信息。设置version="v2",stream_mode=['updates', 'messages']可用统一的v2模式同时流式获取agent进程以及LLM token

运行生命周期

概述：after_model拦截点可以在模型之后、工具调用之前运行，这使得他可以在获得模型相应时通过上下文判断是否包含需要用户操作的工具，若存在则可以创建一个具有action_requests和review_cnfigs的HITLRequest类型并发起中断，然后依据HITLResponse执行同意、更改，或合成拒绝调用的ToolMessage并继续运行。

十四、多智能体

概述：多智能体系统协调各个专门组件来处理复杂的工作流程。不过，单个智能体配备合适的（有时是动态的）工具和提示，通常也能取得类似的结果。主要应用场景有下:

  • 上下文管理
  • 分布式开发
  • 并行化

当单个智能体拥有过多的tool并在使用选择时做出糟糕的决策，当任务需要具有广泛上下文的专业知识，或者需要强行执行顺序约束，只有在满足某些条件后才能解锁功能时，多智能体模式就显得尤为重要。多智能体设计的核心是上下文工程，通过上下文工程决定每个智能体能看到哪些信息。

下为构建多智能体系统的主要模式：

  • Subagents: 主代理协调各个子代理作为工具，所有路由通过主代理，由主代理决定何时以及如何调用每个子代理
  • Handoffs：行为根据状态动态变化。工具调用更新状态变量，从而触发路由或者配置更改、切换代理或调整当前代理的工具和提示
  • Skills: 按需加载专业提示和指示。单个代理始终保持控制，并根据需要从skill加载上下文
  • Router: 路由步骤对输入分类，并将其定向到一个或多个专门的代理。最终结果综合组成一个组合响应
  • Custom workflow: 使用LangGraph构建定制化的执行流程

1.Subagents（次级代理商）

概述：中央主代理（通常称为监督器）通过调用子代理作为工具来协调它们。主代理决定调用哪个子代理、提供什么输入以及如何组合结果。子代理是无状态的——它们不记忆过去的交互，所有对话记忆都由主代理维护。

主要特征：

  • 集中控制：所有路由都通过主代理。
  • 没有直接的用户交互：子代理将结果返回给主代理，而不是用户
  • 通过工具调用子代理：子代理通过工具调用。
  • 并行执行：主智能体可以在单回合内调用多个子智能体。

应用场景：

  • 需要处理多个不同的域时（数据库、邮件）
  • 子代理不需要直接与用户进行交互
  • 集中式工作流程控制

基本实现： 将子代理封装为一个工具，主代理可以调用该工具

一些关键点：

  • 同步与异步：子代理的任务阻塞执行还是异步执行
  • 工具模式：每个代理使用一个工具还是设置单个调度工具
  • 指示子代理方式：系统提示、结构约束（json）与基于工具三种方式
  • 子代理输入：仅传入查询或额外传入完整上下文
  • 子代理输出：次级代理结果或完整上下文历史

2.Handoffs（任务移交）

概述：行为根据状态动态变化。核心机制为工具更新一个状态变量，系统读取该变量调整行为（如应用不同的配置或将请求路由到不同的代理）

主要特征：

  • 状态驱动行为：基于状态变量（例如，current_step或active_agent）的行为改变
  • 基于工具的转换：工具更新状态变量以在状态之间移动。
  • 直接用户交互：每个状态的配置直接处理用户消息。
  • 保持状态：状态在对话轮次中保持不变

应用场景：

  • 需要强制执行顺序约束
  • 客服人员需要不同状态下与用户直接对话、
  • 构建多阶段对话流程

基本实现： 核心机制是一个工具，用于返回更新状态的Command 触发进一步操作。通过给ToolRuntime设置参数（[工具执行时需要的静态上下文的数据类型(如特定的数据库连接)，特定状态参数(继承了AgentState的状态触发类，内含用于判断是否启用某些功能的字段)]）

多智能体子图(LangGraph内容)： 图中的代理以独立节点的形式组成，可以通过Command.PARENT指定接下来要执行的节点 (Graph内容，暂不详细记录)

上下文工程(LangGraph内容) 概述：通过子图切换可以精确控制上下文的流通(Graph内容，暂不详细记录)

多智能体注意事项

  • 上下文过滤系统：每个客户端接收到的上下文的完整性
  • 工具语义：是否仅更新路由状态，是否执行其他操作
  • token效率：平衡上下文完整性和token成本。对话时间延长后，传递的上下文会增多，引发token成本问题

skill（技能）

概述：技能框架中，专业能力被打爆成可调用的“技能”，用于增强智能体的行为。技能主要是由提示驱动的专业化能力。

主要特征：

  • 提示驱动型专业化：技能主要由特定提示定义。
  • 渐进式披露：技能根据上下文或用户需求逐步可用
  • 团块分布：不同团块可以独立发展和保持技能。
  • 轻量级组成：技能比完整的子代理更简单
  • 参考知识：技能可以参考脚本、模板和其他资源。

应用场景： 需要具备多个专业技能的单个智能体，或不同团块需要独立开发相应能力。例如针对不同语言的编码助手、针对不同领域技能的知识库等

基本实现： 在file或database中实现skill体，通过将其调用方式封装在工具中注入智能体。

扩展方式：

  • 动态工具注册：与状态管理结合
  • 层级式技能：树状结构定义技能，创建嵌套的专业划分支。例如加载”数据科学“后可以生成“pandas”、“可视化”、“统计分析”等子技能，从而实现领域知识的细粒度渐进式披露，这种方式可以将能力按逻辑分组，从而有助于管理大型技能知识库
  • 参考信息感知：在技能的提示部分添加指向其他资源的信息，提供这些资源何时可以使用，这样可以增加智能体获取相关资源的方式。

4.router（路由）

概述：通过路由步骤对输入进行分类，将其定向至专门的代理。这对于处理多个高垂直行业知识时非常有用。

主要特征：

  • 路由器分解查询
  • 并行调用零个或多个专门代理。
  • 将结果综合成一个连贯的响应

基本实现： 使用定向用于分析输入信息的LLM对输入信息进行分类总结，然后将分类后的信息块通过Send并行分发给对应的agent。

无状态vs有状态（是否保留之前请求的记忆和上下文）：

  • 例子：用户问“今天天气如何？”，路由器判断这是“天气类”问题，分发给天气 Agent。下一秒用户问“那明天呢？”，如果路由器是无状态的，它可能忘了上一句是在问天气，可能会把“那明天呢？”错误地分发给通用聊天 Agent，因为它缺少上下文。
  • 例子：用户问“今天天气如何？”，路由器分发给天气 Agent。用户接着问“那明天呢？”，有状态的路由器会读取之前的状态（知道刚才在聊天气），从而正确地将“那明天呢？”继续分发给天气 Agent，或者理解为“明天的天气”。

完整信息保持： 如果你希望系统能记住之前的对话（即“有状态”），不要试图让路由器函数本身在内存中保存变量（因为路由器可能是无状态的、分布式的，重启就会丢失数据）。 当路由器决定要把任务交给某个智能体（Agent）时，它的第一步动作是从刚才提到的持久化存储中读取相关的历史消息。

十五、RAG

LLM存在两个主要局限性：

  • 上下文有限-无法一次性摄取整个语料库
  • 静态知识库-训练数据冻结在某个时间点

通过查询时抓取相关外部信息可以解决这个问题

构建知识库： 知识库是指被用于检索过程中的文档库和结构化数据 LangChain的语义搜索技术

从检索到RAG： 检索使得LLM可以在运行时获取相关上下文，实际应用会更进一步：将检索与生成结合，生成给予上下文的答案，这便是RAG的核心思想

检索管道：

构建模块：

  • 文件加载器：从外部来源获取数据，返回标准化Document对象
  • 文本切割器：将大型文档拆分成较小的块，以便可以单独检索并适应模型的上下文窗口
  • 嵌入模型：将文本转换为信息向量
  • 向量存储：存储信息向量
  • 检索

2-step RAG 检索步骤始终在生成步骤之前执行。

Agentic RAG 智能检索增强生成（RAG）结合了检索增强生成和基于智能体的推理的优势。

Hybrid RAG 结合了两步 RAG 和代理 RAG 的特点，引入了查询预处理、检索验证和生成后检查等中间步骤：

  • 查询优化：修改输入的问题以提高检索质量。这可能包括重写不清晰的查询、生成多个变体或使用更多上下文信息扩展查询。
  • 检索验证：评估检索到的文档是否相关且充分。如果不符合要求，系统可能会优化查询并重新检索。
  • 答案验证：检查生成的答案是否准确、完整，以及是否与原文内容一致。如有需要，系统可以重新生成或修改答案。

十六、长期记忆

概述：长期记忆可以跨线程一直存在，且可以随时调用。长期记忆的创建基于LangGraph store,能够将数据保存在按命名空间和键组织的JSON文档。

基本使用： 注册InMemoryStore()对象后注入create_agent的store参数，后续可使用工具调用runtime.store进行读取与写入

存储：每个namespace都以自定义的命名空间（类似于文件夹）和唯一的key（类似于文件名）进行组织。命名空间通常包含用户或组织 ID 或其他标签，以便于组织信息。这种结构支持对内存进行层级组织。
